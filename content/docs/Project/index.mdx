## 核心概念

### 版本控制
<Callout type="success" title="语义化版本控制">
  采用语义化版本控制 2.0.0 标准，确保版本号的语义清晰和一致性。
</Callout>

- **主版本号**：不兼容的API修改
- **次版本号**：向下兼容的功能性新增
- **修订号**：向下兼容的问题修正

### 分支管理
<Callout type="info" title="分支策略">
  采用Git Flow和GitHub Flow的最佳实践，确保团队协作的一致性和代码管理的规范性。
</Callout>

- **特性分支**：新功能开发
- **修复分支**：紧急问题修复
- **发布分支**：版本发布准备
- **实验分支**：技术实验和原型开发

### 提交规范
<Callout type="warn" title="提交信息规范">
  遵循Conventional Commits规范，确保提交信息的可读性和自动化处理能力。
</Callout>

- **格式标准**：`<type>(<scope>): <subject>`
- **类型分类**：feat, fix, docs, style, refactor等
- **自动化验证**：通过husky + commitlint强制校验

## 最佳实践

### 1. 版本发布流程

1. **功能开发** → 在特性分支上开发新功能
2. **代码审查** → 提交Pull Request进行审查
3. **测试验证** → 通过所有自动化测试
4. **版本发布** → 创建发布分支，更新版本号
5. **部署上线** → 部署到生产环境并监控

### 2. 分支管理策略

- **主分支保护**：禁止直接推送到master分支
- **代码审查**：所有合并必须通过Pull Request
- **自动化测试**：必须通过CI/CD检查
- **定期清理**：及时删除已合并的分支

### 3. 提交信息规范

- **使用命令语气**：如"Fix"而非"Fixed"
- **首字母不大写**：保持小写格式
- **不加句号结尾**：简洁明了
- **描述要具体**：说明做了什么，而不是为什么做

## 常见问题

### Q: 如何选择合适的版本号递增方式？
A: 根据变更类型决定：
- 不兼容变更 → 主版本号递增
- 新功能 → 次版本号递增
- Bug修复 → 修订号递增

### Q: 如何处理紧急修复？
A: 
1. 从master分支创建fix分支
2. 快速修复问题并测试
3. 合并到master分支进行紧急部署
4. 同时合并到develop分支保持同步

### Q: 如何写好的提交信息？
A: 
1. 用命令语气（如"Fix"而非"Fixed"）
2. 首字母不大写
3. 不加句号结尾
4. 描述要具体，说明做了什么
